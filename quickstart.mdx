---
title: "Quickstart: build and run grdpwasm in five minutes"
description: "Clone the repository, build the WASM binary and proxy server with a single command, and connect to a remote Windows desktop from your browser."
---

This guide walks you through every step from a fresh checkout to a live remote desktop session in your browser. The whole process takes about five minutes on a machine with Go 1.24 or later already installed.

## Prerequisites

Before you start, make sure you have the following:

- **Go 1.24 or later** — run `go version` to check.
- **Git** — to clone the repository.
- **A reachable RDP server** — a Windows host or any RDP-compatible machine accessible from the machine where you will run the proxy.
- **A modern browser** — any browser with WebAssembly support. For hardware-accelerated H.264 decoding, use Chromium 94+, Edge 94+, or a browser that supports the WebCodecs API.

## Steps

<Steps>
  <Step title="Clone the repository">
    Clone the grdpwasm repository and enter the project directory:

    ```bash
    git clone https://github.com/nakagami/grdpwasm.git && cd grdpwasm
    ```
  </Step>

  <Step title="Build everything">
    Run the default make target to compile all three output artifacts:

    ```bash
    make all
    ```

    `make all` produces:

    | Output | Description |
    |---|---|
    | `static/main.wasm` | Go WASM binary — the full RDP client, runs in the browser. |
    | `static/wasm_exec.js` | Go runtime JavaScript support file, copied from your local Go toolchain. |
    | `proxy/proxy` | WebSocket-to-TCP proxy binary and static file server. |

    <Tip>
      To rebuild individual artifacts, use `make wasm`, `make proxy`, or `make wasm_exec` separately.
    </Tip>
  </Step>

  <Step title="Start the proxy">
    Start the proxy server, which also serves the static WASM files:

    ```bash
    ./proxy/proxy -listen :8080 -static static
    ```

    Alternatively, run:

    ```bash
    make serve
    ```

    Both commands do the same thing. The proxy listens on port 8080, serves `static/` as the web root, and handles WebSocket upgrade requests at `/ws`.

    | Flag | Default | Description |
    |---|---|---|
    | `-listen` | `:8080` | Address and port to listen on. |
    | `-static` | `static` | Directory to serve static files from. |
  </Step>

  <Step title="Open the browser">
    Navigate to [http://localhost:8080](http://localhost:8080) in your browser.

    The page loads the WASM binary and shows **Ready** in the status bar once the module is initialised. If you see **WASM load failed**, check that `static/main.wasm` and `static/wasm_exec.js` both exist.
  </Step>

  <Step title="Connect to a remote desktop">
    Fill in the connection form in the toolbar:

    | Field | Description |
    |---|---|
    | **Host** | Hostname or IP address of the RDP server. |
    | **Port** | RDP port — default `3389`. |
    | **Domain** | Windows domain name. Leave blank for local accounts. |
    | **User** | Username on the remote machine. |
    | **Password** | Password for the account. |
    | **Width / Height** | Initial desktop resolution in pixels (e.g. `1280` × `800`). |

    Click **Connect**. The status bar shows `Connecting…` and then `Connected` (or `Connected (WebCodecs H.264)` if your browser supports the WebCodecs API).
  </Step>
</Steps>

## What you should see

Once connected, the remote desktop appears in the canvas area below the toolbar. You can interact with it immediately using your mouse. The **Connect** button changes to **Disconnect** — click it to end the session cleanly.

## Usage notes

<Note>
  **Keyboard focus:** If keystrokes stop reaching the remote desktop, click directly on the canvas to recapture keyboard focus. The browser tab must be active and the canvas element must have focus for key events to be forwarded.
</Note>

<Note>
  **Remote audio:** Audio from the remote session is streamed via RDPSND and played through the browser's Web Audio API (PCM 44100 Hz, stereo, 16-bit). Audio starts automatically when you click **Connect** — the click gesture is required to satisfy the browser's autoplay policy.
</Note>

<Note>
  **H.264 hardware decoding:** If your browser supports the WebCodecs API, H.264 frames are decoded by the GPU. The status bar shows `Connected (WebCodecs H.264)` when hardware decoding is active, or `Connected` when falling back to software bitmap rendering. Both modes produce a correct remote desktop; WebCodecs is faster for video-heavy workloads.
</Note>
